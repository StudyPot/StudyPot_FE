# 코드 컨벤션

## 개요

이 문서는 이 프로젝트에서 코드를 작성할 때 따르는 기본 규칙을 정리합니다.

- 기술 스택은 `Vue 3`, `TypeScript`, `Vite`, `Tailwind CSS`를 기준으로 합니다.
- 포맷팅은 `Prettier`, 정적 검사는 `ESLint`와 `Oxlint`를 기준으로 합니다.
- 개인 취향보다 프로젝트 전체 일관성을 우선합니다.

## 기본 원칙

- 읽기 쉬운 코드를 우선합니다.
- 한 파일은 한 가지 책임에 집중하도록 작성합니다.
- 중복보다 재사용 가능한 구조를 우선하되, 과한 추상화는 피합니다.
- 코드가 짧아지는 것보다 의미가 명확한 이름을 우선합니다.
- 규칙이 애매할 때는 기존 코드 스타일과 현재 문서 기준을 따릅니다.

## 네이밍

- 변수와 함수 이름은 `camelCase`를 사용합니다.
- 컴포넌트 파일 이름은 `PascalCase`를 사용합니다.
- 타입, 인터페이스, enum 이름은 `PascalCase`를 사용합니다.
- 상수는 일반 상수라면 `camelCase`, 전역 의미의 고정값이라면 `UPPER_SNAKE_CASE`를 사용할 수 있습니다.
- 불리언 변수는 가능하면 `is`, `has`, `can`, `should` 같은 접두어로 시작합니다.

예시:

```tsx
const isLoading = false
const hasPermission = true
function fetchUserProfile() {}
type UserProfile = {
  id: number
  name: string
}
```

## 파일과 폴더

- 컴포넌트는 역할 기준으로 분리합니다.
- 공통으로 재사용되는 코드는 재사용 가능한 위치로 이동합니다.
- 테스트 파일은 대상 파일과 가까운 위치에 두거나 `__tests__` 폴더 규칙을 따릅니다.
- 설정 파일과 문서 파일은 루트 또는 `docs/`에 일관되게 배치합니다.

## Vue 컴포넌트 작성 규칙

- `Vue SFC`는 기본적으로 `<script setup lang="ts">`를 사용합니다.
- 템플릿에서는 Vue 문법에 맞게 `className` 대신 `class`를 사용합니다.
- 컴포넌트는 가능한 한 작고 명확하게 유지합니다.
- 복잡한 계산은 템플릿에 직접 쓰기보다 `computed`나 함수로 분리합니다.
- 템플릿에서 비즈니스 로직이 길어지면 컴포저블 또는 유틸 함수로 분리합니다.

권장 순서:

1. `script`
2. `template`
3. `style`

## TypeScript 사용 규칙

- 가능한 한 `any` 대신 구체적인 타입을 사용합니다.
- 타입 전용 import는 `import type`으로 작성합니다.
- 함수 인자와 반환값은 문맥상 충분히 명확하지 않다면 타입을 명시합니다.
- 선언 파일이 아닌 일반 코드에서는 불필요한 타입 우회를 피합니다.
- 사용하지 않는 값은 제거하고, 의도적으로 사용하지 않는 인자는 `_` 접두어를 사용합니다.

예시:

```tsx
import type { RouteRecordRaw } from 'vue-router'

function formatDate(date: Date): string {
  return date.toISOString()
}
```

## Import 규칙

- import는 파일 상단에 모아 작성합니다.
- 외부 라이브러리 import와 로컬 import는 구분합니다.
- 같은 종류의 import는 가능한 한 정리된 순서로 유지합니다.
- 사이드 이펙트 import는 일반 import 아래에서 명확하게 보이도록 둡니다.

예시:

```tsx
import { createApp } from 'vue'
import { createPinia } from 'pinia'

import App from './App.vue'
import router from './router'
import './style.css'
```

## 스타일 작성 규칙

- 스타일은 우선 `Tailwind CSS` 유틸리티 클래스를 사용합니다.
- 반복되는 스타일 조합이 많아지면 컴포넌트 분리 또는 공통 스타일 전략을 검토합니다.
- 인라인 스타일은 꼭 필요한 경우에만 사용합니다.
- 색상, 간격, 정렬은 임의값 남발보다 일관된 스케일을 우선합니다.
- 컴포넌트 로컬 스타일이 정말 필요할 때만 `<style scoped>`를 사용합니다.

## 상태 관리와 라우팅

- 전역으로 공유할 필요가 없는 상태는 컴포넌트 내부에 둡니다.
- 여러 화면이나 여러 컴포넌트에서 공유되는 상태만 `Pinia`로 올립니다.
- 라우터 관련 값은 URL, 라우트 메타, 네비게이션 책임에 맞게 사용합니다.
- 전역 상태와 라우팅 상태의 책임을 섞지 않습니다.

## 주석 작성 규칙

- 코드만으로 의도가 명확하면 주석을 생략합니다.
- 주석은 "무엇을 하는지"보다 "왜 이렇게 하는지"를 설명하는 데 사용합니다.
- 나중에 수정될 가능성이 큰 설명을 장황하게 적지 않습니다.
- TODO는 실제 후속 작업이 있을 때만 남기고, 가능하면 맥락을 함께 적습니다.

예시:

```tsx
// 서버 응답 스키마가 고정되기 전까지 임시로 기본값을 보정한다.
```

## 조건문과 함수 작성 규칙

- 중첩이 깊어지면 빠른 반환(`early return`)을 우선 고려합니다.
- 함수는 한 가지 역할을 수행하도록 유지합니다.
- 매개변수가 너무 많아지면 객체 인자나 분리를 검토합니다.
- 분기문 안에서 같은 로직이 반복되면 공통 처리로 추출합니다.

## 테스트 작성 규칙

- 새로운 로직이 추가되면 가능한 범위 안에서 테스트를 함께 고려합니다.
- 테스트 이름은 동작이 드러나게 작성합니다.
- 구현 세부사항보다 사용자 관점의 결과나 함수 결과를 검증합니다.
- 테스트용 데이터와 목 객체는 읽기 쉽게 최소한으로 유지합니다.

## 금지 또는 지양 항목

- 사용하지 않는 변수, import, 함수 방치
- 의미가 불분명한 축약어 남발
- 불필요한 `any` 사용
- 템플릿 내부의 과도한 로직 작성
- 포맷팅 수동 수정 후 `Prettier` 미적용
- 디버깅용 `console.log`, `debugger`를 정리하지 않고 커밋

## 작업 전 확인 사항

- 새 코드가 기존 프로젝트 구조와 어울리는지 확인합니다.
- 포맷팅은 `Prettier` 기준으로 맞춥니다.
- 정적 검사는 `ESLint` 기준으로 통과시키는 것을 기본으로 합니다.
- 문서화가 필요한 규칙 변화가 있으면 `docs/` 문서도 함께 갱신합니다.
