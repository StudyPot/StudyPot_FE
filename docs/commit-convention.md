# 커밋 컨벤션

## 개요

이 프로젝트는 현재 Git 히스토리 기준으로 아래 형식의 커밋 메시지를 사용합니다.

```txt
[TYPE] 작업 설명
```

예시:

- `[BUILD] tailwind css 적용`
- `[BUILD] prettier 설정`
- `[FIX] html 언어 ko 설정`
- `[CHORE] prettier format 설정`
- `SPT-123 [FEAT] 회원가입 페이지 추가`

## 기본 규칙

- 타입은 영어 대문자와 대괄호 형식으로 작성합니다.
- 설명은 한국어로 간결하게 작성합니다.
- 한 커밋에는 한 가지 목적만 담는 것을 원칙으로 합니다.
- 가능한 한 커밋 메시지만 봐도 무엇을 했는지 바로 이해되도록 작성합니다.
- 불필요하게 긴 문장보다 짧고 명확한 표현을 우선합니다.

## 형식

기본 형식:

```txt
[TYPE] 작업 설명
```

Jira 브랜치에서 Git hook이 적용된 경우 실제 커밋 메시지는 아래 형식으로 저장됩니다.

```txt
SPT-123 [TYPE] 작업 설명
```

작성 예:

```txt
[BUILD] tailwind css 적용
[FIX] 로그인 버튼 비활성화 오류 수정
[CHORE] prettier ignore 파일 추가
SPT-123 [FEAT] 회원가입 페이지 추가
```

## 타입 기준

### `[BUILD]`

빌드, 개발 환경, 설정 파일, 도구 체인 관련 변경에 사용합니다.

예:

- `Tailwind CSS` 설정
- `Prettier` 설정
- `ESLint` 설정
- `Vite`, `PostCSS`, `TypeScript` 설정 변경
- 의존성 추가 또는 빌드 관련 스크립트 수정

예시:

- `[BUILD] tailwind css 적용`
- `[BUILD] eslint 설정`

### `[FIX]`

버그 수정이나 잘못된 설정값 수정에 사용합니다.

예:

- 화면 동작 오류 수정
- 잘못된 속성값 수정
- 접근성/SEO 메타데이터 수정
- 오타나 잘못된 로직 수정

예시:

- `[FIX] html 언어 ko 설정`

### `[CHORE]`

기능 변화는 없지만 유지보수성이나 작업 편의성을 높이는 변경에 사용합니다.

예:

- 포맷 스크립트 정리
- ignore 파일 추가
- 개발용 정리 작업
- 리팩터링까지는 아닌 소규모 구조 정리

예시:

- `[CHORE] prettier format 설정`

## 필요 시 사용할 수 있는 타입

현재 히스토리에는 없지만, 아래 타입도 필요하면 사용할 수 있습니다.

### `[FEAT]`

사용자 기능 추가에 사용합니다.

예:

- 새 페이지 추가
- 새 컴포넌트 기능 추가
- API 연동 기능 추가

예시:

- `[FEAT] 회원가입 페이지 추가`

### `[DOCS]`

문서 추가 또는 문서 수정에 사용합니다.

예:

- `README` 수정
- `docs/` 문서 추가
- 가이드 문서 정리

예시:

- `[DOCS] prettier 컨벤션 문서 추가`

### `[REFACTOR]`

기능 변화 없이 내부 구조를 개선할 때 사용합니다.

예:

- 함수 분리
- 컴포넌트 구조 개선
- 중복 로직 정리

예시:

- `[REFACTOR] 라우터 설정 분리`

### `[TEST]`

테스트 추가나 테스트 코드 수정에 사용합니다.

예:

- 단위 테스트 추가
- Playwright 시나리오 보강
- 테스트 데이터 정리

예시:

- `[TEST] App 컴포넌트 렌더링 테스트 추가`

## 타입 선택 기준

- 기능을 추가했다면 `BUILD`보다 `FEAT`를 우선합니다.
- 설정 파일을 바꿨더라도 목적이 버그 수정이면 `FIX`를 사용합니다.
- 사용자 기능 변화 없이 개발 환경이나 보조 설정만 바꿨다면 `BUILD` 또는 `CHORE`를 사용합니다.
- 문서만 수정했다면 `DOCS`를 사용합니다.

## 추천 기준

- 도구 설정, 빌드 설정, 의존성 연결: `[BUILD]`
- 동작 오류 수정, 잘못된 값 수정: `[FIX]`
- 포맷, ignore, 관리성 개선: `[CHORE]`
- 사용자 기능 추가: `[FEAT]`
- 문서 추가/수정: `[DOCS]`

## Jira 브랜치 사용 시

- 브랜치 이름에 Jira 이슈 키가 있으면 커밋 메시지 앞에 같은 키가 자동으로 붙습니다.
- 예: `SPT-123/signup-page`
- 예: `SPT-123 [FIX] 로그인 버튼 비활성화 오류 수정`
- 팀원이 같은 동작을 쓰려면 한 번 `npm install` 또는 `npm run setup:hooks`를 실행해야 합니다.

## 커밋 작성 원칙

- 커밋 하나에는 가능한 한 하나의 주제만 담습니다.
- 설정 변경과 기능 추가는 가능하면 분리합니다.
- 포맷 변경만 있는 경우 다른 로직 변경과 섞지 않는 것을 권장합니다.
- 나중에 `git log`만 봐도 작업 흐름이 보이도록 메시지를 작성합니다.
